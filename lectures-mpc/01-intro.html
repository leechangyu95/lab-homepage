1. 왜 MPC인가 — 한 줄 비유

레이싱 카로 시작하기

F1 드라이버가 코너를 어떻게 도는지 떠올려 보세요. 핸들을 단순히 "지금 차가 코스 가운데에서 얼마나 벗어났는가"만 보고 꺾지 않습니다. 앞에 어떤 코너가 있고, 다른 차가 어디 있고, 노면이 어떤지를 미리 살피고 — 짧은 미래의 주행선을 머릿속에 그려놓고 한 발자국씩 옮기죠. 갑자기 차가 끼어들면 그 새 정보를 반영해 다시 그립니다.

이게 바로 모델 예측 제어 (Model Predictive Control, MPC)의 본질입니다. "미래를 짧게 시뮬레이션 → 가장 좋은 입력 시퀀스를 찾기 → 그중 첫 입력만 적용 → 한 스텝 후 측정값으로 다시 처음부터" 이 4단계 루프가 매 샘플 시간마다 반복됩니다.

MPC의 한 줄 정의 — 매 샘플 시간마다 유한 미래에 대한 최적 제어 문제를 새로 풀어, 그 첫 입력만 적용하는 제어. 줄여서 "Plan & Do".

왜 이런 방식이 필요한가?

전통적 PID는 "지금 오차가 얼마인가"에만 반응합니다. 빠르고 단순하다는 큰 장점이 있지만 두 가지 한계가 있습니다.

  • 미래의 정보를 못 씁니다. 알고 있는 외란(예보된 날씨, 예측된 도로 곡률)이 있어도 PID는 "현재 오차"만 봅니다.
  • 제약을 직접 다루지 못합니다. 액추에이터 한계(±10 V), 안전 제약(온도 ≤ 80℃), 상태 제약(충돌 회피)을 PID에 정식으로 넣을 방법이 없어서 보통 안티 와인드업이나 외부 saturation 같은 임시방편으로 막습니다.

MPC는 이 둘을 정면으로 해결합니다. 미래 N 스텝의 시스템 모델을 들고 최적화 문제를 풀어 "어떤 입력 시퀀스가 비용을 최소화하면서 모든 제약을 만족하는가"를 답합니다.

🎮 인터랙티브: Plan & Do (수신지평선 전략)

차(파랑)가 곡선 도로를 따라 목표(빨강)로 향합니다. 매 스텝마다 호라이즌 N만큼 앞을 내다보고 점선 경로를 미리 계획한 뒤, 그 첫 한 스텝만 실행합니다. 다음 스텝에서 다시 새 측정값으로 처음부터 계획해요. 속도 슬라이더로 시뮬레이션 속도를 바꿔보세요.

2. 고전 제어 vs MPC — 두 관점

같은 "플랜트를 원하는 응답으로 만들기"라는 목표를 두 학파가 전혀 다른 도구로 풀어왔습니다.

관점고전 제어 (PID, Lead-Lag)MPC
설계 영역주파수 영역 (Bode, Root Locus)시간 영역 (상태공간)
주요 관심외란 제거, 잡음 둔감, 모델 불확실성제약 처리 (입력·상태·안전)
설계물오프라인에 컨트롤러 $C(s)$를 한 번 짜둠매 샘플마다 온라인으로 최적화
제약 처리임시방편 (saturate, anti-windup)최적화 제약으로 정식 포함
다입출력SISO 위주, MIMO는 디커플링 어려움MIMO 자연스럽게 처리
계산 비용매우 가벼움 (덧셈·곱셈 몇 번)매 스텝마다 QP/NLP 풀기

둘은 적이 아니라 상보적입니다. 빠른 내부 루프(예: 모터 전류)에는 PID, 외부 계획·조정에는 MPC를 쓰는 계층 구조(cascade)가 자율차·드론·산업 공정의 표준 패턴입니다.

3. 제약이 MPC를 필요하게 만드는 이유

모든 실제 시스템에는 제약이 있다

  • 물리적 제약 — 액추에이터 한계 (모터 ±12V, 밸브 0~100%, 비행기 받음각 ±15°)
  • 성능 제약 — 오버슈트 ≤ 5%, 정착시간 ≤ 2초
  • 안전 제약 — 원자로 온도 ≤ 임계값, 차량 ≤ 차선 경계

그리고 가장 중요한 사실: 최적의 동작점은 보통 제약 근처에 있습니다. 자동차는 타이어가 미끄러지기 직전까지 가속해야 가장 빠르고, 화학 공정은 안전 한계 직전에서 가장 효율적이며, 제약에서 멀리 떨어져 보수적으로 운전하면 비용·효율을 잃습니다.

고전 제어의 접근: 안전을 위해 동작점을 제약에서 충분히 떨어뜨려 운전 → 보수적·비효율.
MPC의 접근: 제약을 최적화 문제에 그대로 넣어 → 제약을 침범하지 않는 한도 내에서 가장 좋은 입력을 자동으로 계산.

🎮 인터랙티브: 제약이 있을 때 PID vs MPC

1차 시스템 $\dot x = -x + u$를 목표 $r=10$으로 보내려 합니다. 단, 입력 제한 $|u|\le u_{\max}$. $u_{\max}$를 줄이면 PID는 와인드업으로 오버슈트가 커지는 반면, MPC는 제약을 알고 미리 부드럽게 감속해요.

4. 수학적 정식화 — 최적화 문제로

식 한 줄로 보기

매 시점 $t$에서 측정 $x(t)$를 받고, 다음 유한 호라이즌 최적화 문제를 풉니다:

$$ U_t^*(x(t)) := \arg\min_{U_t} \sum_{k=0}^{N-1} q(x_{t+k}, u_{t+k}) $$ $$ \text{subj. to}\quad x_t = x(t)\ \text{(측정)} $$ $$ x_{t+k+1} = A x_{t+k} + B u_{t+k}\ \text{(시스템 모델)} $$ $$ x_{t+k} \in \mathcal{X},\quad u_{t+k} \in \mathcal{U}\ \text{(상태·입력 제약)} $$ $$ U_t = \{u_t, u_{t+1}, \dots, u_{t+N-1}\}\ \text{(최적화 변수)} $$

요소 하나씩:

  • 비용 $q(x, u)$ — "원점에서 떨어진 거리", "에너지 사용량", "오차 제곱" 등. 가장 흔한 선택은 이차형 $q = x^\top Q x + u^\top R u$.
  • 모델 $x_{k+1} = Ax_k + Bu_k$ — 미래를 예측하는 "내부 시뮬레이터". 선형이면 LMPC, 비선형이면 NMPC.
  • 제약 $\mathcal{X}, \mathcal{U}$ — 안전·성능·물리적 한계. 부등식 또는 폴리헤드론 형태로 표현.
  • 호라이즌 $N$ — 미래를 몇 스텝 내다볼지. 길수록 정확하지만 계산 비용도 비례.

이차 비용 LMPC — 가장 흔한 형태

실무에서 가장 많이 쓰는 형태는 이차 비용 + 선형 모델 + 선형 제약입니다. 이러면 문제가 이차계획법(Quadratic Programming, QP)이 되어 잘 알려진 솔버(OSQP, qpOASES, quadprog)로 ms 단위에 풀 수 있습니다.

$$ \min_{U,X} \sum_{k=0}^{N-1}(x_k^\top Q x_k + u_k^\top R u_k) + x_N^\top P x_N $$ $$ \text{s.t.}\quad x_{k+1} = A x_k + B u_k,\quad u_{\min} \le u_k \le u_{\max},\quad x_{\min} \le x_k \le x_{\max} $$

$Q \succeq 0$, $R \succ 0$, $P \succeq 0$. 행렬 $P$는 종단 비용으로, 안정성 보장에 핵심 역할을 합니다 (Part III 주제).

5. 수신지평선 전략 (Receding Horizon)

왜 첫 입력만 쓰고 나머지는 버리나?

최적화로 $N$개 입력을 한꺼번에 풀어놓고, 그중 첫 번째 $u_t^*$만 시스템에 적용한 뒤 나머지 $u_{t+1}^*, \dots, u_{t+N-1}^*$는 버립니다. 다음 스텝에서 새 측정값 $x(t+1)$로 처음부터 다시 풉니다.

"다 풀어놓고 버리다니 낭비 아닌가?"라고 생각할 수 있지만, 이게 MPC를 강건하게 만드는 핵심입니다.

  • 외란·모델 오차에 즉각 반응 — 측정값이 예측과 다르면 다음 스텝에서 즉시 새 계획.
  • 피드백 효과 — 매 스텝마다 측정값을 다시 받아 들어가니 사실상 폐루프.
  • 실현 가능성 보호 — 미래에 예상치 못한 일이 생겨도 더 늦지 않게 재계획.

6. 설계 파라미터 — 호라이즌·가중치

호라이즌 $N$ — 얼마나 멀리 볼까

  • 너무 짧으면 — 근시안. 미래 제약이나 외란을 못 봐서 충돌·오버슈트.
  • 너무 길면 — 계산 비용이 $\mathcal{O}(N)$~$\mathcal{O}(N^3)$로 증가, 모델 오차가 복리로 누적.
  • 실용 가이드 — 시스템의 정착시간 $T_s$의 1~2배에 해당하는 스텝 수. 자율차는 1~3초, 빌딩 HVAC은 12~24시간.

가중치 $Q, R$ — 빠름 vs 부드러움

이차 비용 $\sum (x^\top Q x + u^\top R u)$에서 $Q$는 "오차에 얼마나 민감", $R$은 "입력 사용에 얼마나 인색".

  • $Q$ ↑, $R$ ↓ → 빠른 응답, 큰 입력, 진동·오버슈트 가능
  • $Q$ ↓, $R$ ↑ → 부드러운 입력, 느린 응답, 정적 오차

실용적으로는 $R$을 1로 고정하고 $Q$의 대각 원소를 상태별로 조정합니다 ($Q = \text{diag}(q_1, \dots, q_n)$). 각 $q_i$는 "그 상태가 1만큼 틀렸을 때의 비용"입니다.

🎮 인터랙티브: 가중치 $Q/R$ 비율의 효과

같은 1차 시스템을 같은 호라이즌으로 제어하지만 $Q/R$ 비율을 바꿉니다. 비율이 크면 빨리 가지만 입력이 거칠고, 작으면 부드럽지만 느려요.

8. MPC의 장점과 도전과제

장점

  • 제약 처리 일원화 — 안티 와인드업, saturate, override 같은 임시방편 없이 정식으로 다룸.
  • 다입출력 자연스러움 — MIMO 시스템도 SISO처럼 단일 최적화 문제로 표현.
  • 예보 정보 활용 — 알려진 미래 외란(날씨, 도로 곡률, 가격)을 모델에 넣을 수 있음.
  • 성능 한계까지 운전 가능 — 제약 근처에서 안전하게 동작해 효율 극대화.

도전과제

  • 실시간 계산 — 매 샘플 시간 안에 QP/NLP를 풀어야 함. 빠른 시스템(ms 단위)에서는 explicit MPC, 코드 생성, 가속 솔버 필수.
  • 안정성 — 폐루프 안정성이 자동 보장되지 않음. 종단 비용 $P$, 종단 제약 $\mathcal{X}_f$를 잘 설계해야 함 (Part III 주제).
  • 강건성 — 모델 오차·외란에 민감할 수 있음. 강건 MPC, 튜브 MPC, 확률적 MPC 등 변형으로 대응.
  • 가용성(feasibility) — 어떤 시점에서 모든 제약을 만족하는 입력이 존재하지 않을 수 있음. 소프트 제약(slack), 회복 모드로 처리.

  • OSQP / qpOASES — 빠른 QP 솔버. 선형 MPC에 표준.
  • do-mpc (Python) — 입문용 고수준 인터페이스, 산업 응용에 적합.

10. 코드 예제 — 1D MPC (Python + CVXPY)

import numpy as np
import cvxpy as cp

# 시스템: x_{k+1} = A x_k + B u_k (이산화된 1차)
A, B = 0.9, 0.1
N = 15           # 호라이즌
Q, R = 1.0, 0.05
x0 = 0.0
ref = 10.0
u_max = 3.0

def solve_mpc(x0):
    x = cp.Variable(N+1)
    u = cp.Variable(N)
    cost = 0
    cons = [x[0] == x0]
    for k in range(N):
        cost += Q*(x[k] - ref)**2 + R*u[k]**2
        cons += [x[k+1] == A*x[k] + B*u[k]]
        cons += [cp.abs(u[k]) <= u_max]
    cost += Q*(x[N] - ref)**2     # 종단 비용
    prob = cp.Problem(cp.Minimize(cost), cons)
    prob.solve()
    return u.value[0]              # 첫 입력만 반환

# 수신지평선 루프
x = x0
for t in range(50):
    u_apply = solve_mpc(x)
    x = A*x + B*u_apply            # 시스템 업데이트
    print(f"t={t:2d}  x={x:6.3f}  u={u_apply:6.3f}")

11. 핵심 정리

  • MPC = 매 샘플마다 미래 N스텝 최적화 → 첫 입력만 적용 → 반복 (Plan & Do).
  • 고전 제어가 못 다루는 제약과 다입출력을 정면으로 처리.
  • 이차 비용 + 선형 모델 + 선형 제약 → QP, ms 단위 해결 가능.
  • 호라이즌 $N$은 정착시간의 1~2배, $Q/R$ 비율로 응답 속도/부드러움 트레이드오프.
  • 실시간성·안정성·강건성·가용성 4대 도전과제는 Part II/III에서 깊이 다룸.
  • 응용은 ms(드론) ~ days(빌딩)까지 같은 프레임워크.
