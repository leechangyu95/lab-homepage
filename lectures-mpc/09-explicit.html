Week 9. Explicit MPC

매 샘플마다 QP를 푸는 대신, 오프라인에 미리 다 풀어 lookup table로 저장해두면 어떨까요? 이게 Explicit MPC의 출발점입니다. Bemporad 등(2002)이 제안한 multi-parametric programming의 결과로, 작은 임베디드 시스템에서 ms 이하 응답이 가능해집니다.

목차

  1. 왜 Explicit MPC인가 — 임베디드의 한계

1. 왜 Explicit MPC인가

온라인 MPC의 한 사이클은:

  1. 현재 상태 $x(t)$ 측정
  2. QP 풀이 — 가장 비싼 부분 (수십 μs ~ 수 ms)
  3. 첫 입력 $u_0^*$ 적용

임베디드 마이크로컨트롤러(Cortex-M4 등)에서 QP는 일반적으로 ms 단위 — 빠른 시스템(전력 변환 1 MHz, 자세 제어 1 kHz)에는 너무 느림. Explicit MPC는 이 QP를 단순한 "룩업 + 행렬 곱"으로 대체합니다.

2. Multi-parametric QP (mp-QP)

표준 MPC의 QP는 매 샘플마다 다음 형태:

$$ U^*(x) = \arg\min_U\ \tfrac{1}{2} U^\top H U + x^\top F^\top U,\ \text{s.t.}\ G U \le w + S x $$

핵심 관찰: $x$는 매번 바뀌지만 $H, F, G, w, S$는 상수. 즉 이 문제는 $x$를 매개변수로 한 QP 패밀리입니다. 이를 Multi-parametric QP라 부릅니다.

해 $U^*(x)$를 $x$의 함수로 한 번에 표현할 수 있다면, 매 샘플마다 그 함수에 $x$를 대입하기만 하면 됨!

3. 해의 구조 — 조각별 affine

놀라운 결과 (Bemporad·Morari·Dua·Pistikopoulos, 2002):

핵심 정리 — 다면체 제약 + 이차 비용 mp-QP의 최적해 $U^*(x)$는 $x$의 조각별 affine 함수이고, 정의 영역은 다면체로 분할된 임계 영역(critical region)들의 합집합입니다.

구체적으로:

$$ U^*(x) = F_i x + g_i,\quad x \in \mathcal R_i,\ i = 1, \ldots, M $$

각 $\mathcal R_i$는 다면체 (선형 부등식 다발), $(F_i, g_i)$는 그 영역에서 활성화된 제약 패턴에 의해 결정되는 행렬·벡터.

왜 그럴까? 각 $x$에 대해 KKT 조건이 활성 제약 집합 $\mathcal A(x)$를 결정합니다. 같은 $\mathcal A$를 갖는 $x$들의 집합이 하나의 다면체 영역이고, 그 안에서 KKT 조건은 선형 방정식 — 해는 $x$의 affine 함수.

2D 예시

1D 적분기 $x_{k+1} = x_k + u_k$, $|u_k| \le 1$, 호라이즌 $N=2$. 상태 공간이 2D (현재와 한 스텝 전 상태). 해를 그려보면:

  • 중심부 — 제약 비활성, $u^* = -K x$ (LQR)
  • 오른쪽 — $u_0 = -1$ 활성
  • 왼쪽 — $u_0 = +1$ 활성
  • 가장자리 — 두 입력 모두 포화

경계는 직선들이고, 각 영역에서 해는 $x$의 다른 affine 함수.

4. 오프라인 알고리즘 — 활성 집합 열거

mp-QP를 풀어 모든 $(F_i, g_i, \mathcal R_i)$를 얻는 절차:

  1. 임의의 내부점 $x_0$에서 QP 풀이 → 활성 집합 $\mathcal A_0$ 결정.
  2. $\mathcal A_0$가 정의하는 임계 영역 $\mathcal R_0$ 와 affine 해 $(F_0, g_0)$ 계산.
  3. $\mathcal R_0$의 각 면(facet)을 가로질러 인접 영역 탐색 → 새 활성 집합 → 새 영역.
  4. 모든 영역이 발견될 때까지 BFS/DFS 반복.

결과는 "이진 탐색 트리 + 영역별 affine 해" lookup table. 도구: MATLAB MPT3 (Multi-Parametric Toolbox)가 표준.

5. 온라인 평가

실시간으로는 두 단계만:

  1. 영역 찾기 (Point Location) — 현재 $x$가 어느 $\mathcal R_i$에 속하는지 결정.
  2. 해 평가 — $u_0 = (F_i x + g_i)$의 첫 원소.

영역 찾기 가속

  • 순차 탐색 — 모든 $\mathcal R_i$를 차례로 시험. $O(M)$, 작은 문제 OK.
  • 이진 탐색 트리 — 분리 초평면으로 트리 구성 → $O(\log M)$.
  • 해시 테이블 — 정해진 격자에 해를 캐싱.

현대 임베디드에서는 BST + AVX/SIMD로 영역 찾기를 μs 단위에 끝냄. 자동차 ESC, 전력 변환 같은 high-speed 응용에서 표준.

6. 한계 — 차원의 저주

장점이 분명한 만큼 한계도 분명:

  • 영역 수 폭발 — 상태 차원 $n$, 호라이즌 $N$, 제약 수 $q$가 커지면 영역 수 $M$이 지수적으로 폭발. 일반적으로 $n \le 5$, $N \le 10$ 정도가 실용 한계.
  • 저장 공간 — 각 영역의 다면체 표현 + affine 해 행렬이 메모리 차지. 큰 문제는 MB ~ GB.
  • 비선형 불가 — NMPC는 mp-QP의 우아한 구조가 깨지므로 직접 적용 어려움.
  • 모델 변경 시 재계산 — 모델 식별·재튜닝마다 오프라인 mp-QP 다시 돌려야 함.

그래서 Explicit MPC는 "작은 시스템 + 빠른 응답이 절대 필요"한 응용에 한정. 큰 시스템은 OSQP 같은 빠른 온라인 솔버 + 워밍 스타트가 표준.

🎮 인터랙티브: 1D Explicit MPC — 영역 분할

1D 적분기에 호라이즌 $N=3$, 입력 제약 $|u| \le 1$인 MPC. 각 점은 해가 어느 활성 패턴에 속하는지를 색으로 보여줍니다. 커서를 움직여 각 영역의 affine 해를 확인.

7. 핵심 정리

  • Explicit MPC = QP의 해를 $x$의 조각별 affine 함수로 미리 풀어둔 것.
  • 오프라인 mp-QP로 영역 + affine 행렬 계산. 도구: MPT3.
  • 온라인은 영역 찾기(BST $O(\log M)$) + 행렬 곱 한 번 → μs 단위.
  • 장점: 매우 빠름, 결정적, 검증 가능. 작은 임베디드에 최적.
  • 한계: 영역 수 폭발 → $n \le 5$, $N \le 10$ 정도가 실용 한계.
  • 큰 시스템은 OSQP + 워밍 스타트 같은 빠른 온라인 솔버가 더 효율적.

📚 깊이 1 — Explicit MPC가 등장한 이유

온라인 QP 풀이가 무거운 응용 (μs 단위 임베디드, 전력변환기 등)에서는 매 시점 풀이가 비현실적. Explicit MPC는 이를 해결합니다.

  • 오프라인: 모든 가능한 상태에 대한 최적 입력을 미리 계산.
  • 온라인: 현재 상태가 어느 영역에 속하는지 lookup → 미리 저장된 affine 함수 평가.
  • 풀이 시간: ns~μs (lookup + 곱셈만).

📚 깊이 2 — Multi-Parametric Quadratic Programming (mpQP)

QP의 비용·제약이 파라미터 $x$에 affine하게 의존:

$$\min_z \frac{1}{2}z^THz + x^TFz \quad\text{s.t.}\quad Gz \le W + Sx$$

이 문제의 최적해 $z^*(x)$는:

  • 상태 공간을 다면체 영역(critical region)으로 분할.
  • 각 영역 안에서 $z^*(x) = K_i x + k_i$ (조각 affine).
  • 비용 $V^*(x)$는 조각 이차형 (PWQ).

이게 explicit MPC의 핵심.

📚 깊이 3 — 친절 비유

Explicit MPC = 메뉴판. 매번 주방에서 새로 요리(QP 풀이)하는 대신, 모든 손님 유형(상태)에 대한 답을 미리 메뉴판에 적어둠. 손님 오면 어느 유형인지 보고 그대로 서빙(lookup).

📚 깊이 4 — Critical Region (CR)

같은 active constraint set을 갖는 상태들의 집합. $x$가 이 영역 안에 있으면 동일한 affine 게인 사용.

  • 각 CR은 다면체 (선형 부등식의 교집합).
  • CR들이 상태 공간을 분할 (서로 겹치지 않고 합집합 = feasible region).
  • CR 수가 게인의 다양성을 결정.

📚 깊이 5 — CR 수의 폭발

일반적으로 CR 수는 호라이즌 $N$, 제약 수, 상태 차원에 지수적으로 증가.

📚 깊이 6 — 단계별 예제 — 1차원 explicit MPC

$x_{k+1} = x_k + u_k$, $|u| \le 1$, $N = 1$, $Q = R = 1$.

Unconstrained: $u^*(x) = -K x$ where $K$ from LQR ≈ 0.618.
제약 $|u| \le 1$:
  • $x \le -1/K$: $u^* = +1$ (상한).
  • $-1/K \le x \le 1/K$: $u^* = -Kx$ (자유).
  • $x \ge 1/K$: $u^* = -1$ (하한).
3개 CR. 각 영역 안에서 affine 함수.

📚 깊이 7 — 온라인 평가 알고리즘

  1. 현재 상태 $x$ 측정.
  2. $x$가 어느 CR에 속하는지 결정 (binary tree, 직접 검색 등).
  3. 해당 영역의 affine 함수 $u = K_i x + k_i$ 평가.
  4. 적용.

📚 깊이 8 — Lookup 알고리즘

  • Sequential search: 모든 CR 순차 점검. $O(N_{CR})$.
  • Binary search tree: 미리 구축. $O(\log N_{CR})$.
  • Kd-tree: 공간 분할.
  • Hash 기반: 그리드 + lookup.

📚 깊이 11 — Approximate Explicit MPC

큰 차원에서도 Explicit를 가능하게 하는 방법:

  • NN(Neural Network)로 $u^*(x)$ 함수 근사.
  • 피드포워드 네트워크가 PWA 함수 표현 가능 (ReLU).
  • 최근 deep learning 접근.
  • 이론적 안정성·feasibility 보장은 추가 작업 필요.

📚 깊이 12 — 응용

  • 전력 변환기 (FCS-MPC) — 100kHz 가능.
  • 모터 드라이브.
  • 드론 자세 (작은 모델).
  • 기타 μs 단위 응용.

📚 깊이 13 — 흔한 오답

  1. 큰 차원 시스템에 Explicit 적용 → 메모리 폭발.
  2. CR 검색 알고리즘 미적용 → 평가 시간 증가.
  3. 온도·파라미터 변동 무시 → 다시 계산 필요.
  4. Soft 제약 처리 누락.

📚 깊이 15 — FAQ

Q1. Explicit이 항상 좋은 선택?

아니요. 차원 작고 빠른 응답 필요할 때만. 차원 크면 implicit + warm start.

Q2. CR 수를 줄이는 방법?

호라이즌 단축, 제약 단순화, 차원 축소.

Q3. NN이 Explicit MPC를 대체?

일부 응용에서 가능. 그러나 안정성 보장은 활발한 연구 분야.
