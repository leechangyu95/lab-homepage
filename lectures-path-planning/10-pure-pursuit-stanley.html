1. Guidance 가 뭐길래

한 줄 비유부터

친구 차에 동승해서 코스를 도는 모습을 떠올려 보세요. 운전석엔 세 사람의 역할이 합쳐져 있습니다. (1) 머릿속에 코스 전체를 그린 네비게이터 — "200 m 가서 좌회전", (2) 매 순간 "지금은 저 빨간 콘 쪽으로 핸들을 살짝 꺾어"라고 짚어주는 코치, (3) 실제로 핸들과 페달을 조작하는 운전자. 자율주행에서 이 셋이 각각 경로계획(W6–W7), 유도(이번 주), 제어(W9 PID)입니다.

왜 PID 만으로는 안 되는가

PID 는 "오차를 0 으로 만드는 법"은 알지만 "오차를 어떤 기준으로 잴지"는 모릅니다. 그 기준을 정해주는 것이 유도. Pure Pursuit, Stanley, PN 은 결국 "오차의 정의"가 다른 세 가지 유도법입니다.

  • 경로계획(Planning, W6–W7) — 점들의 나열 $\{p_1, \ldots, p_N\}$. 시간 개념 없음.
  • 유도(Guidance, 이번 주) — 매 순간 상태 $x(t)$ 를 보고 "지금 어디를 향할지"를 출력. 실시간 갱신.
  • 제어(Control, W9) — 유도가 준 목표를 모터·핸들로 실제 실행.

실차에선 보통 다른 주기로 동작합니다 — 계획 1 Hz, 유도 20–50 Hz, 제어 100–1000 Hz. 빠른 쪽이 위 단계의 출력을 받아 실시간으로 결정합니다.

2. Pure Pursuit — 강아지가 먹이 쫓듯

한 줄 비유부터

강아지에게 막대를 멀리 던지면, 강아지는 막대까지 일직선으로 가지 않습니다. 현재 머리가 향한 방향에서 막대 쪽으로 호를 그리며 달려가요. 머리를 살짝 돌리고 한 발 내딛고, 또 살짝 돌리고… Pure Pursuit 이 정확히 그 동작입니다.

구체적으로는 차의 뒷바퀴축에서 일정 거리 $L_d$ 앞에 있는 경로상의 점(lookahead point)을 골라, 차가 그 점에 도달하는 호를 그리도록 조향각을 계산합니다.

왜 일직선이 아니라 호인가

자동차는 비홀로노믹입니다(W2). 옆으로 직접 못 가고, 핸들을 꺾어 호를 그릴 수밖에 없어요. 그래서 "현재 위치 → 목표점"을 잇는 가장 자연스러운 자동차 궤적은 호(원호)이고, Pure Pursuit 은 그 호의 곡률에 맞는 조향각을 한 번에 계산합니다.

식 한 줄씩 풀어보기

핵심 식은 정확히 한 줄입니다.

$$ \boxed{\;\delta = \arctan\!\left(\frac{2 L \sin\alpha}{L_d}\right)\;} $$
  • $L$ — 차의 휠베이스(앞·뒷바퀴축 사이 거리). W2 자전거 모델 그림을 떠올리세요.
  • $\alpha$ — 차체 헤딩과 "차 → lookahead 점" 사이 각도. "얼마나 옆을 봐야 하는가".
  • $L_d$ — lookahead 거리. 차에서 경로상 목표점까지의 거리.
  • $\delta$ — 우리가 구하려는 앞바퀴 조향각.

기하 유도 — 왜 이 식이 나오는가

종이에 그림 하나 그리면 풀립니다.

  1. 차의 뒷바퀴를 점 $A$, lookahead 점을 점 $B$. 거리 $|AB| = L_d$.
  2. 차가 그리는 호의 반지름을 $R$ 이라 하면, $A$ 에서 호를 따라 $B$ 에 도달.
  3. 중심·$A$·$B$ 로 이루어진 이등변삼각형에서 $R = \dfrac{L_d}{2\sin\alpha}$.
  4. 자전거 모델: 휠베이스 $L$ 차가 곡률 반경 $R$ 로 회전하려면 $\tan\delta = L/R$.
  5. 둘을 합치면 $\tan\delta = \dfrac{2L\sin\alpha}{L_d}$ → 양변에 $\arctan$.

핵심은 "호 한 개를 기하로 그리는 것"이지 적분이나 최적화가 들어가지 않는다는 점. 그래서 Pure Pursuit 은 임베디드 환경에서도 매우 가볍게 돕니다.

숫자로 직접 풀어보기
휠베이스 $L = 2.5$ m, lookahead $L_d = 6$ m, 각도 $\alpha = 20°$ 일 때 조향각은?
$\sin 20° \approx 0.342$, 따라서
$\delta = \arctan\!\left(\dfrac{2 \cdot 2.5 \cdot 0.342}{6}\right) = \arctan(0.285) \approx 15.9°$.
곡률 반경 $R = L_d / (2\sin\alpha) \approx 8.77$ m.

속도 적응형 lookahead $L_d = k v + L_{fc}$

$L_d$ 를 어떻게 정하느냐가 Pure Pursuit 성능의 거의 모든 것을 좌우합니다. 운전 경험과 정확히 일치해요.

  • $L_d$ 가 작으면 — 가까운 점만 보고 운전. 작은 흔들림이 큰 핸들 조작으로 증폭돼 지그재그(oscillation). 초보가 보닛 끝만 보며 운전하는 모습.
  • $L_d$ 가 크면 — 멀리 보고 운전. 부드럽지만 코너 안쪽을 잘라먹는 cutting corners. 베테랑이 코너 진입 전 미리 핸들을 꺾는 느낌.

실무 표준은 속도 적응형:

$$ L_d = k\, v + L_{fc}, \quad k \in [0.1, 0.2],\ L_{fc} \in [2, 3]\text{ m} $$
$L_d$직선곡선
1 m (짧음)정확지그재그 진동
3 m (중간)양호양호
6 m (길음)부드러움내측 컷
$0.1 v + 2$ (적응형)균형 ✅균형 ✅
실패 모드 — "내 차가 자꾸 흔들려요" — 거의 100% $L_d$ 가 너무 작거나, 차가 너무 빨라 $L_d$ 를 늘려야 할 때입니다. 디버깅 순서: ① $L_d$ 두 배 → 흔들림 감소? lookahead 문제. ② 그래도 그러면 조향 actuator 의 시간지연(latency) 가능성.

3. 인터랙티브: lookahead $L_d$ 의 효과

🎮 Pure Pursuit — Lookahead 와 호 그리기

곡선 경로(초록)를 자전거 모델 차량(파랑)이 추종합니다. 주황 점이 lookahead 점, 점선이 차량과 lookahead 사이 시선, 빨간 화살표가 앞바퀴와 조향각 $\delta$. $L_d$ 를 작게 줄이면 지그재그가, 크게 늘리면 코너 안쪽 자르기가 보입니다. "속도 적응형" 토글을 켜면 $L_d = 0.1 v \cdot \text{scale} + L_{fc}$ 로 자동 조절됩니다.

꼭 만져볼 시나리오

  • $L_d = 25$ + $v = 4$ → 곡선에서 핸들이 휙휙 — 지그재그.
  • $L_d = 170$ → 곡선 안쪽을 잘라먹음 — cutting corners.
  • 속도 적응형 체크 → 속도 슬라이더만 흔들면 $L_d$ 가 자동으로 따라옴. RMSE 가 안정적임을 확인.

코드 — Pure Pursuit (PythonRobotics 단순화)

import numpy as np

WB = 2.9   # 휠베이스 (m)

def pure_pursuit(state, path, Lfc=2.0, k=0.1):
    """
    state = (x, y, yaw, v),  path = ndarray[N, 2]
    return: (delta, target_index)
    """
    x, y, yaw, v = state
    Ld = k * v + Lfc                          # 속도 적응형 lookahead

    # 1) 가장 가까운 경로 점
    dx = path[:, 0] - x
    dy = path[:, 1] - y
    d  = np.hypot(dx, dy)
    i  = int(np.argmin(d))

    # 2) Ld 이상 떨어진 점까지 전진
    while i < len(path) - 1 and d[i] < Ld:
        i += 1
    tx, ty = path[i]

    # 3) 조향각
    alpha = np.arctan2(ty - y, tx - x) - yaw
    delta = np.arctan2(2 * WB * np.sin(alpha), Ld)
    return delta, i

4. Stanley — 횡오차 + 헤딩오차

한 줄 비유부터

자전거를 타고 도로 위 흰 차선을 따라간다고 상상하세요. 차선에서 벗어나면 두 가지를 동시에 신경 씁니다 — (1) "지금 차선에서 얼마나 옆으로 떨어졌나?"(횡방향 오차 $e_{fa}$), (2) "내 자전거가 차선과 같은 방향을 보고 있나?"(헤딩 오차 $\theta_e$). 이 두 오차를 각도로 환산해 더한 값이 핸들을 꺾어야 할 양 — 정확히 Stanley 의 정의입니다.

왜 두 오차를 더하나

  • 횡오차만 보면 — 차선에서 옆으로 1 m 떨어졌는데 차가 차선과 평행하면 핸들을 꺾어 차선 쪽으로 가다가 통과해 반대편으로 1 m 떨어집니다 — 진동.
  • 헤딩오차만 보면 — 차 방향만 차선과 맞추면 그 자리를 그대로 평행 이동만. 영영 안 돌아옴.
  • 둘을 더하면 — 옆으로 떨어졌으니 핸들을 꺾고($\arctan$ 항), 동시에 비스듬한 만큼 보정($\theta_e$ 항). 자연스럽게 매끄럽게 합류.

핵심 식

$$ \boxed{\;\delta = \theta_e + \arctan\!\left(\frac{k\, e_{fa}}{v + v_s}\right)\;} $$
  • $\theta_e = \psi_r - \psi$ — 헤딩 오차. 경로 접선과 차 헤딩의 차.
  • $e_{fa}$ — 차의 앞바퀴 위치에서 잰 횡방향 오차. Stanley 의 핵심 — Pure Pursuit 이 뒷바퀴 기준이라면 Stanley 는 앞바퀴 기준.
  • $k$ — 횡오차 게인(보통 $0.5$).
  • $v_s$ — softening 항. 저속에서 분모 폭발 방지(보통 $1.0$).

속도가 분모에 들어가는 이유 — 빠르게 달릴 때 같은 횡오차에도 핸들 보정량을 자동으로 줄여서 부드럽게 만듭니다. 고속에서 부드럽게, 저속에서 민첩하게 — 속도 보정이 식 안에 내장돼 있어요.

숫자로 직접 풀어보기
$e_{fa} = 0.5$ m, $\theta_e = 5° \approx 0.087$ rad, $v = 10$ m/s, $k = 1$, $v_s = 1$.
$\arctan(1 \cdot 0.5 / 11) \approx 0.045$ rad $\approx 2.6°$, 따라서 $\delta \approx 5° + 2.6° = 7.6°$.
같은 상황 $v = 20$ → 조향각이 $5° + 1.3° = 6.3°$ 로 자동으로 줄어듭니다.

왜 그 이름?

2005 DARPA Grand Challenge 우승차의 이름이 "Stanley"였습니다(Stanford, Thrun 팀, VW 투아렉 베이스). 사막 212 km 를 6시간 54분에 완주했고, 그때 사용된 lateral controller 가 이 식이라 "Stanley method" 라 부릅니다.

코드 — Stanley

def stanley(state, path, yaw_path, k=0.5, vs=1.0):
    x, y, yaw, v = state
    # 전륜 위치
    fx = x + WB * np.cos(yaw)
    fy = y + WB * np.sin(yaw)

    # 가장 가까운 경로 점 (전륜 기준)
    dx = fx - path[:, 0]; dy = fy - path[:, 1]
    d  = np.hypot(dx, dy)
    i  = int(np.argmin(d))

    # Cross-track error (부호 포함)
    yaw_p = yaw_path[i]
    e_fa  = -np.cos(yaw_p) * dy[i] + np.sin(yaw_p) * dx[i]

    # Heading error (wrap 필수!)
    theta_e = wrap(yaw_p - yaw)

    delta = theta_e + np.arctan2(k * e_fa, v + vs)
    return np.clip(delta, -np.deg2rad(30), np.deg2rad(30)), i

def wrap(a):
    return (a + np.pi) % (2 * np.pi) - np.pi

5. 인터랙티브: Pure Pursuit vs Stanley

🎮 같은 S-커브에서 두 기법 동시 비교

파랑 차 = Pure Pursuit, 빨강 차 = Stanley. 같은 속도, 같은 시작점에서 같은 경로(초록)를 따라갑니다. RMSE 가 화면 하단에 누적 표시됩니다. Pure Pursuit 의 $L_d$ 를 키우면 코너에서 밀려나는 게(내측 컷) 보입니다.

관찰 포인트 — 기본값에서 둘 다 비슷한 RMSE 가 나오지만, $v$ 를 4.0 까지 올리고 PP $L_d = 30$ 으로 줄이면 Pure Pursuit 만 진동합니다(RMSE 급증). 반대로 $L_d = 180$ 으로 키우면 PP 가 코너 안쪽을 자르며 Stanley 보다 RMSE 가 커집니다.

6. Proportional Navigation — 미사일이 적기 잡는 법

한 줄 비유부터

친구가 던지는 공을 강아지가 잡는 모습을 떠올려 보세요. 강아지는 공이 지금 있는 곳으로 뛰지 않습니다. 공이 지나갈 자리를 예측해 그쪽으로 달려요. 이 "미래 위치 가로채기"가 바로 Proportional Navigation(PN)입니다.

LOS(Line-of-Sight)와 회전율 $\dot\lambda$

미사일에서 표적을 잇는 직선이 LOS. 그 직선이 시간에 따라 회전하는 각속도가 $\dot\lambda$. 미사일과 표적이 실제로 충돌하려면 — 두 점의 상대 속도가 LOS 와 평행해야 하므로 — $\dot\lambda = 0$ 이어야 합니다.

핵심 식

$$ \boxed{\;a_\perp = N\, V_c\, \dot\lambda\;} \qquad \text{또는 회전 형태로}\quad \dot\psi_M = N\, \dot\lambda $$
  • $a_\perp$ — 미사일에 줄 수직 가속(조향 명령에 해당).
  • $N$ — 항법상수. 보통 3–5. 비조작 표적 요격에서 횡가속 적분 $\int a^2\,dt$ 가 최소가 되는 값이 $N=3$(Bryson–Ho, 1975).
  • $V_c = -\dot R$ — 접근 속도(거리가 줄어드는 비율).
  • $\dot\lambda$ — LOS 회전율.

왜 이게 통하는가 — 병렬 항법(parallel navigation)의 원리입니다. $\dot\lambda \neq 0$ 이면 충돌하지 못한다는 것이 기하적 사실. PN 은 그 회전을 즉시 $N$ 배만큼 강하게 보정해 자동으로 0 으로 끌어당깁니다.

자율주행 차에 직접 PN 을 쓰는 일은 드물지만, 아이디어 — "표적의 미래 위치를 고려해 지금 조향" — 는 여러 곳에서 다시 등장합니다.

7. 인터랙티브: PN 으로 움직이는 표적 요격

🎮 PN(파랑) vs PPG(주황) 동시 발사

같은 시작점·속도·표적에서 두 미사일을 동시에 발사. PPG 는 매 순간 표적의 지금 위치로 돌고, PN 은 LOS 회전율 $\dot\lambda$ 에 비례해 회전합니다. 표적 패턴(직선/사인/급기동)과 항법상수 $N$ 을 만져보세요. PN 이 거의 항상 더 빨리 명중합니다.

  • $N = 3$ — 비조작 표적 요격에서 이론적으로 최적.
  • $N < 2$ — 반응이 너무 느려 표적을 놓침.
  • $N > 5$ — LOS 측정 잡음에도 과민 반응.

코드 — PN guidance

def pn_guidance(ego_state, target_state, N=4):
    ex, ey, evx, evy = ego_state
    tx, ty, tvx, tvy = target_state
    dx, dy = tx - ex, ty - ey
    R      = np.hypot(dx, dy)

    dvx, dvy = tvx - evx, tvy - evy
    # LOS 회전율 λ_dot
    lam_dot = (dx * dvy - dy * dvx) / (R**2 + 1e-9)
    # 접근 속도
    Vc      = -(dx * dvx + dy * dvy) / R
    # 수직 가속 명령
    a_perp  = N * Vc * lam_dot
    return a_perp

8. 실전 통합 패턴 — 횡(Stanley) + 종(PID)

실차에서 가장 흔한 자율주행 구조 — 횡·종 분리. 횡방향은 Pure Pursuit/Stanley, 종방향은 PID:

while running:
    # 1) 횡방향 — Stanley
    delta, i = stanley(state, ref_path, ref_yaws)
    # 2) 종방향 — PID (속도 프로파일은 W11 에서)
    v_ref = speed_profile[i]
    a     = pid_speed.step(v_ref - state[3], dt)
    # 3) 차량에 인가
    car.step(a, delta, dt)

곡률 기반 Feedforward — Pure Pursuit 의 약점 보완

Pure Pursuit 의 내측 컷은 곡률 기반 feedforward 한 줄로 대부분 해결됩니다.

$$ \delta = \underbrace{\arctan\!\left(\frac{2L\sin\alpha}{L_d}\right)}_{\text{Pure Pursuit (FB)}} + \underbrace{\arctan(L\,\kappa)}_{\text{곡률 FF}} $$

이 강의 W11 통합 시뮬에서도 PP + FF 조합을 기본으로 사용합니다.

10. 손풀이 연습 3개

Q1. $v = 10$ m/s, $k = 0.15$, $L_{fc} = 2$ → $L_d$ = ?
풀이: $L_d = 0.15 \cdot 10 + 2 = 3.5$ m.

Q2. Stanley 에서 정지($v = 0$) 시 $v_s = 0$ 이면 어떻게 되나?
풀이: $\arctan(k e_{fa} / 0)$ 으로 발산. $v_s \geq 1.0$ 가 의무.

Q3. 같은 트랙에서 Pure Pursuit 이 내측 컷을 만드는 이유?
풀이: lookahead 점이 곡선 안쪽에 있고, 차가 그 점으로 호를 그리는 동안 실제 경로보다 안쪽 궤적이 됨.

11. 핵심 정리

  • Pure Pursuit: $\delta = \arctan(2L\sin\alpha / L_d)$ — "앞 점 따라가기". 후륜 기준, 한 식, $L_d$ 한 개.
  • Stanley: $\delta = \theta_e + \arctan(k e_{fa}/(v + v_s))$ — "헤딩 맞추기 + 거리 줄이기". 전륜 기준, 속도 보정 내장.
  • PN: $a_\perp = N V_c \dot\lambda$ — "LOS 회전율 죽이기". 움직이는 표적용.
  • 저속 로봇 → PP, 고속 자율주행 → Stanley, 동적 표적 → PN(아이디어 차용).
  • 다음 주(W11)에서 PP + PID + A* + Spline 을 한 시뮬레이터에서 통합.
