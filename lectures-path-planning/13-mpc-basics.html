Week 13. MPC 기초 — 미래를 보고 제어하는 법

PID는 "지금 오차"만 보고 반사적으로 반응합니다. MPC(Model Predictive Control)는 체스 플레이어처럼 몇 수 앞을 예측하고, 그 안에서 가장 좋은 명령을 골라 첫 수만 실행한 뒤 다시 계산합니다. 이번 주는 그 "예측 → 최적화 → 실행" 사이클을 인터랙티브 데모로 직접 굴려봅니다.

1. 한 줄 비유 — 체스 플레이어처럼 두기

초보 체스 플레이어는 "지금 당장 좋은 수"를 둡니다. 고수는 다릅니다 — 지금 둔 수가 5수, 10수 뒤에 어떤 결과로 이어질지 머릿속으로 굴려보고, 그중에서 가장 유리한 첫 수를 둡니다. 다음 차례에는 상대방의 반응을 보고 다시 5~10수를 굴려봅니다. 즉 매 수마다 처음부터 다시 계획합니다.

이게 정확히 MPC가 하는 일입니다. 자율주행 차량에 적용하면:

  • 지금부터 약 3초 후까지 차의 미래 궤적을 모델로 예측한다.
  • 그 3초 동안 줄 수 있는 모든 조향·가속 시퀀스 중 비용이 가장 낮은 것을 푼다.
  • 첫 명령만 차량에 보낸다. 나머지는 버린다.
  • 0.1초가 지나면 새 측정값을 가지고 처음부터 다시 푼다.
한마디로 — PID = 반사 신경, MPC = 전략 사고. PID가 "어이쿠 차가 차선에서 30cm 벗어났네, 보정!" 한다면, MPC는 "2초 뒤에 코너가 오니까 지금 미리 살짝 안쪽으로 붙자"가 가능합니다.

2. 왜 PID로는 부족한가

9주차에 다룬 PID는 단순하고 빠르며 대부분의 상황에서 충분합니다. 하지만 자율주행에는 PID가 자연스럽게 풀지 못하는 두 가지 문제가 있습니다.

문제PID 입장MPC 입장
조향각 한계 $|\delta| \le 30°$출력에 saturation을 후처리로 붙임 → wind-up 위험최적화 제약으로 자연스럽게 포함
곡선 진입 미리 대비오차가 발생한 후에야 보정 시작예측 horizon 안에 곡률이 보이면 미리 조향
차선 경계 $|e_y| \le 1.5$제약을 직접 표현할 방법 없음$x \in \mathcal{X}$로 한 줄
승차감 vs 추종 trade-off$K_p, K_i, K_d$ 시행착오$Q, R$ 가중치로 명시적 설계

3. Receding Horizon — "굴러가는 시야"

4. 인터랙티브 1 — Receding Horizon 시각화

Receding Horizon 차량 — 매 스텝 새 계획, 첫 명령만 실행

1D 위치 시스템 $x_{k+1}=x_k+u_k\Delta t$에 대해 $N$ 스텝 앞 예측 궤적(파란 점선)이 매 스텝마다 어떻게 다시 그려지는지 확인하세요. 빨강은 실제 차량 경로, 주황은 horizon 안에서 실제로 적용된 첫 명령들입니다.

꼭 관찰할 것: horizon $N$을 5에서 25까지 올려보세요. $N$이 작으면 멀리 못 보고 현재만 따라가는 PID 같은 거동, $N$이 크면 미리 부드럽게 수렴합니다. 단 계산 비용은 $N$에 비례해서 늘어납니다.

5. 수식으로 보는 MPC 3 기둥

모든 MPC 문제는 다음 세 가지로 정의됩니다.

① 모델 (Prediction Model)

현재 상태 $x_k$와 입력 $u_k$가 주어졌을 때 다음 상태를 어떻게 예측할지.

  • 선형 시불변(LTI): $\;x_{k+1} = A\,x_k + B\,u_k$
  • 선형 시변(LTV): $\;x_{k+1} = A_k\,x_k + B_k\,u_k$ — 매 스텝 행렬 갱신
  • 비선형: $\;x_{k+1} = f(x_k, u_k)$ — Nonlinear MPC, 더 무겁다

② 목적 함수 (Cost)

"무엇을 최소화할 것인가". 보통 추종 오차 + 제어 노력의 가중합입니다.

$$J = \sum_{k=0}^{N-1}\Big(\|x_k - x_{ref}\|_Q^2 + \|u_k\|_R^2\Big) + \|x_N - x_{ref,N}\|_{Q_f}^2$$

  • $Q$ — 상태 오차 페널티. 크게 잡으면 목표 추종 강조.
  • $R$ — 입력 페널티. 크게 잡으면 부드러운 조향 강조.
  • $Q_f$ — 종단 비용. horizon 끝에서의 상태가 무리하지 않도록 강하게 유지.

③ 제약 (Constraints)

  • 모델: $x_{k+1} = A x_k + B u_k$ (등호 제약)
  • 입력 한계: $u_{\min} \le u_k \le u_{\max}$ (조향각, 가속도)
  • 입력 변화율: $|u_k - u_{k-1}| \le \Delta u_{\max}$ (부드러운 조향)
  • 상태 한계: $x_k \in \mathcal{X}$ (차선 폭, 속도 한계)

이 세 가지를 매 제어 주기마다 풀어 $u_0^*$를 얻고, $u_0^*$만 차량에 인가합니다. 다음 주기에는 새 상태로 처음부터 다시 풉니다.

6. 인터랙티브 2 — Q · R 가중치 튜닝

$Q$ vs $R$ trade-off — 같은 차량, 같은 곡선, 다른 가중치

$Q$를 크게 하면 차가 곡선을 더 정확히 따라가지만 조향이 거칠어집니다. $R$을 크게 하면 조향은 부드러워지지만 차선 안쪽으로 잘려나가는 컷 인(cut-in) 현상이 생깁니다. 슬라이더를 만져 trade-off를 체감해보세요.

실험 가이드
  • $Q=50, R=0.1$: 정확하지만 조향이 한계(빨간 점선)에 자주 부딪힘.
  • $Q=2, R=20$: 조향 부드러움, 그러나 RMSE 큼.
  • $Q=10, R=5$ 정도가 보통의 시작점입니다.

7. 제약 조건 — 자연스럽게 반영하는 무기

MPC의 가장 큰 매력은 "물리적 한계"를 코드 한 줄로 표현할 수 있다는 점입니다.

# 조향 각도 한계
cons += [cp.abs(u[k]) <= np.deg2rad(30)]

# 조향 변화율 (부드러운 운전)
cons += [cp.abs(u[k] - u[k-1]) <= np.deg2rad(60) * dt]

# 차선 경계 (lateral error 1.5 m 이내)
cons += [cp.abs(z[k, 0]) <= 1.5]

# 종방향: 가속·감속 한계, 속도 한계
cons += [a[k] >= -5.0, a[k] <= 3.0]
cons += [v[k] >= 0.0,  v[k] <= 15.0]

PID로는 이런 조건을 일일이 후처리(saturation, anti-windup, limit guard)로 막아야 했지만, MPC는 최적화 단계에서 처음부터 만족시키는 해를 찾습니다.

8. 인터랙티브 3 — 제약 ON/OFF, MPC vs LQR

제약이 있을 때 MPC와 LQR이 갈리는 순간

같은 1D 시스템에 같은 비용을 주었습니다. MPC는 조향(입력) 한계 $|u| \le u_{max}$를 인지하고 미리 분배합니다. LQR은 한계를 모르고 무한대 입력을 가정해 푼 뒤 후처리로 잘립니다 — 그 결과 오버슈트와 정착 지연이 생깁니다.

"목표 반전" 버튼을 눌러 큰 step을 만들고 $u_{max}$를 조이세요. LQR은 saturate 영역에서 windup 비슷한 지연을 보입니다. MPC는 한계 안에서 미리 페이스를 맞춥니다.

9. 1D MPC를 손과 코드로 풀어보기

손풀이 — 제약 안에서 가장 빨리 닿기

다음 문제를 같이 풀어봅시다.

  • 모델: $x_{k+1} = x_k + u_k$, $\Delta t = 1$
  • 현재: $x_0 = 0$, 목표 $x_{ref} = 5$
  • $N = 3$, 제약 $|u| \le 2$
  • 비용: $J = \sum (x_k - 5)^2 + 0.1\,u_k^2$

제약이 없다면 $u_0 = 5$ 한 번이면 끝이지만, $|u| \le 2$ 때문에 한 스텝에 5는 못 갑니다. 따라서 제약 안에서 최대한 빠르게:

  • $u_0 = 2 \Rightarrow x_1 = 2$
  • $u_1 = 2 \Rightarrow x_2 = 4$
  • $u_2 = 1 \Rightarrow x_3 = 5$ — 도달 후 바로 명령을 줄여 $R$ 항을 절약

cvxpy로 똑같이 풀기

import cvxpy as cp
import numpy as np

def mpc_1d(x0, x_target, N=10, dt=0.1, u_max=2.0):
    x = cp.Variable(N + 1)
    u = cp.Variable(N)
    cost = 0
    cons = [x[0] == x0]
    for k in range(N):
        cost += cp.square(x[k] - x_target) + 0.1 * cp.square(u[k])
        cons += [x[k+1] == x[k] + u[k] * dt]
        cons += [cp.abs(u[k]) <= u_max]
    cost += 10 * cp.square(x[N] - x_target)        # terminal cost
    cp.Problem(cp.Minimize(cost), cons).solve(solver=cp.OSQP)
    return float(u.value[0]), x.value, u.value

# Receding horizon 실행
state = 0.0
log = []
for t in range(50):
    u0, x_pred, u_pred = mpc_1d(state, x_target=5.0, N=10, dt=0.1)
    state = state + u0 * 0.1
    log.append((state, u0))
왜 첫 명령만 쓰는가? — horizon $u_0,\dots,u_{N-1}$ 전체가 다 풀려있는데도 $u_0$만 적용하는 이유는, 다음 스텝에는 측정값이 갱신되어 더 정확한 정보로 다시 풀 수 있기 때문입니다. 미리 계산된 $u_1$은 "이전 측정 기반"이라 신선도가 떨어집니다.

10. 자동차 횡방향 MPC

자율주행에서 가장 흔한 형태인 오차 좌표(error frame) 횡방향 MPC를 봅시다. 상태 $\mathbf{z} = [e_y,\,e_\psi]^\top$ (lateral / heading error), 입력 $\delta$ (조향), 외부 신호 $\kappa_r$ (참조 곡률).

$$ \begin{bmatrix}e_{y,k+1}\\e_{\psi,k+1}\end{bmatrix} = \begin{bmatrix}1 & v\Delta t\\0 & 1\end{bmatrix} \begin{bmatrix}e_{y,k}\\e_{\psi,k}\end{bmatrix} + \begin{bmatrix}0\\v\Delta t/L\end{bmatrix}\delta_k + \begin{bmatrix}0\\-v\Delta t\end{bmatrix}\kappa_{r,k} $$

def mpc_lat(z0, kappa_ref, v=10.0, L=2.9, dt=0.1, N=20):
    z = cp.Variable((N + 1, 2))
    u = cp.Variable(N)
    A = np.array([[1, v*dt], [0, 1]])
    B = np.array([0, v*dt/L])
    C = np.array([0, -v*dt])

    Q  = np.diag([10.0, 2.0])
    R  = 5.0
    Qf = np.diag([50.0, 10.0])

    cost = 0
    cons = [z[0] == z0]
    for k in range(N):
        cost += cp.quad_form(z[k], Q) + R * u[k]**2
        cons += [z[k+1] == A @ z[k] + B * u[k] + C * kappa_ref[k]]
        cons += [cp.abs(u[k]) <= np.deg2rad(30)]
    cost += cp.quad_form(z[N], Qf)
    cp.Problem(cp.Minimize(cost), cons).solve(solver=cp.OSQP)
    return float(u.value[0])
지표Pure Pursuit (W10)MPC (오늘)
직선 RMSE0.08 m0.03 m
곡선 최대 $|e_y|$0.30 m0.08 m
조향 saturate발생없음
계산 시간< 1 ms~ 5 ms

곡선에서 차이가 큰 이유는 명확합니다 — Pure Pursuit는 "지금 한 점만 보고" 조향각을 결정하지만, MPC는 horizon 안에 들어온 곡률을 미리 보고 진입 전부터 차의 자세를 만들어둡니다.

11. 실용 팁 — Warm Start, LTV, 안전 결합

Warm Start로 실시간성 확보

매 스텝 QP를 처음부터 푸는 것은 낭비입니다. 이전 스텝의 해를 초기 추측으로 주면 OSQP 같은 솔버는 몇 ms 안에 수렴합니다.

prob.solve(solver=cp.OSQP, warm_start=True)

LTV-MPC — 속도가 변하면 행렬도 변한다

$v$가 시간에 따라 변하면 $A, B$도 매 스텝 갱신해야 합니다.

for k in range(N):
    v_k = v_ref[k]
    A_k = np.array([[1, v_k*dt], [0, 1]])
    B_k = np.array([0, v_k*dt/L])
    cons += [z[k+1] == A_k @ z[k] + B_k * u[k] + C_k * kappa_ref[k]]

이 구조가 양산 ADAS에서 가장 흔히 쓰입니다.

MPC + CBF — 안전과 성능을 동시에

지난 주(W12) 본 CBF 조건을 MPC 제약으로 바로 추가할 수 있습니다.

for k in range(N):
    cons += [cbf_condition(z[k], u[k], obstacles)]   # 12주차 h(x) >= 0

이렇게 하면 "예측 기반 고성능 추종 + 증명 가능한 안전"이라는 2024–2026 자율주행의 핵심 트렌드가 완성됩니다.

실차 주의 — Python + cvxpy 그대로는 100 Hz 제어가 어렵습니다. 실시간 임베디드에는 acados나 FORCES Pro 같이 C 코드를 자동 생성해주는 툴체인을 쓰는 것이 표준입니다.

12. 핵심 정리 & 흔한 실수

Take-aways
  • MPC = 모델 + 비용 + 제약을 매 스텝 풀어 첫 명령만 적용하는 receding horizon 제어.
  • $Q$ 키우면 정확, $R$ 키우면 부드럽다. 시작값 $Q=\text{diag}(10,2),\,R=5,\,Q_f=5Q$.
  • MPC가 곡선에서 강한 이유는 "예측 horizon이 곡률을 미리 본다"는 한 줄.
  • $N=20{-}30$, $\Delta t=0.05{-}0.1$이 자율주행 횡제어의 표준 (총 2–3초 예측).
#실수예방
1전체 시퀀스 그대로 실행첫 명령만 적용 후 다시 풀기
2$Q, R$에 음수/0 사용모두 양수, 대각 행렬 권장
3warm start 미사용으로 느림OSQP warm_start=True
4제약이 너무 tight → infeasibleslack 변수로 soft constraint
5속도 변화에도 LTI 모델 고정매 스텝 LTV로 재선형화
